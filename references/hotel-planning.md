# 酒店规划方法

> 本文件是 SKILL.md Step 2 酒店候选的详细方法论。
> 配套：`./amap-mcp-usage.md` 场景 3（查酒店）。

---

## 1. 总体原则

### 1.1 酒店即锚点

- 每天行程必须落在酒店 **30 分钟通勤圈内**（**优先地铁/公交圈**，非驾车圈）
- 否则提示用户换酒店
- 30 分钟是"舒适边界"，不是"硬上限"
- **选点排序**：地铁口 500m 内 > 公交枢纽 > 仅打车方便

### 1.2 少即是好

- 单城行程：**1 个酒店**（除非有强理由）
- 多城行程：**每城 1 个**
- 一城换两酒店的成本 > 一家稍贵但全程顺

### 1.3 选酒店 ≠ 订酒店

本 skill 只**推荐酒店候选** + **生成跳转链接**。
**不代替 Booking / Agoda / 携程** 之类的 OTA 完成预订。

## 2. 用户输入（Step 1 必问项）

如果用户说"酒店待选 / 帮我推荐酒店"，问这些字段：

| 字段 | 选项 |
|------|------|
| **预算** | A. < ¥500/晚 B. ¥500-1000 C. ¥1000-2000 D. ¥2000+ E. 不限 |
| **类型** | A. 商务连锁（稳） B. 精品设计（拍照） C. 民宿 / Airbnb（本地感） D. 传统日式 / 和风 E. 不挑 |
| **区域偏好** | A. 跟着行程主区域 B. 交通枢纽附近 C. 安静的住宅区 D. 网红区 |
| **早餐** | A. 含早 B. 不含 |
| **特殊需求** | 接送 / 行李寄存 / 24h 前台 / 洗衣 / 健身房 |

**四拍格式**：

> **Re-ground**：行程规划 Step 2，酒店候选。
> **Simplify**：先确认预算和类型，免得推荐一堆用户不会订的。
> **Recommend**：默认推商务连锁 + 跟着行程主区域，性价比最高。
> **Options**：A. 商务 / B. 精品 / C. 民宿 / D. 和风

## 3. 单城酒店候选流程

### 3.1 区域锚定

先用 V1 验证逻辑跑一遍 Step 3 草案，确定每天主区域：

```
Day 1 主区域：新宿
Day 2 主区域：浅草
Day 3 主区域：镰仓
Day 4 主区域：富士山（远郊）
Day 5 主区域：机场方向
```

**酒店选址策略**：选**覆盖天数最多**的区域。

```
新宿：Day 1 ✅ + Day 2(中转) ✅ + Day 3(中转) ✅ + Day 4(出发) ✅ + Day 5(返程) ✅ = 5/5
浅草：Day 1(到不了) ❌ + Day 2 ✅ + Day 3 ❌ + Day 4 ❌ + Day 5 ❌ = 1/5
```

**新宿胜出**。

### 3.2 关键词搜索

```
工具：poi_search
输入：
  keywords: "酒店"
  city: "东京"
  location: "新宿站经纬度"
  radius: 1000（米，步行 15 分钟）
输出：pois[]
```

**取前 10-20 条**（按距离 / 评分排序）。

### 3.3 三类候选筛选

从 Top 10-20 里**人工或 AI 选** 3 个：

| 类型 | 选择标准 |
|------|---------|
| **商务连锁** | 评分 ≥ 4.3 / 价格在中位 / 品牌（Dormy Inn / APA / 东横 INN 等） |
| **精品设计** | 评分 ≥ 4.5 / 有设计标签 / 价格偏高 / 拍照友好 |
| **民宿 / 公寓** | 评分 ≥ 4.7 / 含厨房 / 价格灵活 / 长住友好 |

### 3.4 通勤圈校验（V1 衍生）

对每个候选酒店跑：

```
对每天主区域中心 → 该酒店：
  distance 或 route_transit 实算通勤时间
  任何一天 > 30 分钟 → 警告
```

**输出格式**：

```
## 候选酒店 A：新宿王子大饭店
- 区域：新宿歌舞伎町
- 价格：约 ¥800/晚
- 类型：商务连锁
- 通勤：
  - Day 1 主区域新宿：步行 5 分钟 ✅
  - Day 2 主区域浅草：地铁 25 分钟 ✅
  - Day 3 主区域镰仓：JR 30 分钟 ⚠️（临界）
  - Day 4 主区域富士山：新干线 1 小时 50 分钟 ✅（远郊日不可避免）
  - Day 5 主区域机场：成田特快 1 小时 20 分钟 ✅
- 小红书笔记：隔音一般 / 近歌舞伎町夜生活 / 早餐不错
```

### 3.5 用户确认

把 3 个候选 + 通勤表给用户看，让用户选：

> A. 新宿王子大饭店（商务，¥800）
> B.  sequence 旅馆（精品，¥1500）
> C. Airbnb 公寓（民宿，¥600）

用户选完后进入 Step 3 主体规划。

## 4. 多城行程酒店候选

### 4.1 拆分策略

```
行程：东京 3 天 + 京都 2 天
酒店数：2（东京 1 个 + 京都 1 个）
```

### 4.2 跨城交通

选酒店时**必须同时考虑**跨城交通：

```
东京酒店：
  离东京站近（去京都坐新干线方便）
  vs
  离成田 / 羽田近（Day 5 返程方便）

推荐：
  东京站附近酒店（兼顾两端，机场靠特快直达）
```

### 4.3 多酒店带来的复杂度

每多一个酒店，V1-V7 验证要多跑一次。建议：

- **3 城以内**：每城 1 个酒店
- **4-5 城**：考虑减少城市数（参考 V1 删除清单）
- **> 5 城**：强烈建议砍到 4 城以内

## 5. 酒店决策的反模式

| 反模式 | 后果 |
|--------|------|
| 只看价格不看位置 | 通勤时间爆炸 |
| 只看评分不看价格 | 超预算 |
| 选网红拍照酒店 | 隔音差 / 交通不便 |
| 选"机场附近"酒店 | 每天进城花 1 小时 |
| 多酒店为了"体验不同" | 通勤混乱，行李搬运行 |
| 跨城酒店选在中间（不在两边） | 两边都跑 |

## 8. 极端情况处理

### 8.1 用户预算极低（< ¥300/晚）

- 推青旅 / 胶囊旅馆
- 注意通勤（往往偏远）
- 牺牲部分体验换预算

### 8.2 用户预算极高（> ¥3000/晚）

- 推顶级酒店 + 服务
- 强调体验而非省钱
- 注意：很多奢华酒店离市中心远，反而通勤差

### 8.3 带小孩 / 老人

- 强调房间面积 / 早餐 / 接送
- 避免换酒店
- 牺牲区域自由度换便利度

### 8.4 长住（> 7 天）

- 推 Airbnb / 公寓
- 推带厨房的（节省餐费）
- 强烈建议单酒店
