# 餐厅调研方法（v1.2.0 重构：双轨零装 + 深度档）

> **v1.2.0 重大变化**：原方案把大众点评当成"国内餐厅硬信号主力"，但 `verify.meituan.com` 反爬极严（签名 + 行为指纹 + 滑块），所有**直接抓大众点评 M 站**或 **Playwright 跑主站**的方案都**必败**。opencli + Browser Bridge + 真实 Chrome session 是唯一可绕过反爬的方案，但需要：
> - 装 npm 包 + 装 Chrome 扩展 + 装 Chrome + 扫码登录
> - 维护成本高（Chrome 必须常开、不能换电脑、不能多人）
> - 反爬升级就挂
>
> **新方案（v1.2.0）**：**餐厅硬信号走"高德 POI 详情" + "美团攻略（WebFetch）"双轨，零装零扫码**。opencli 降级为"深度档"——只给"非要拿必吃榜 + 真实评价数 + 排队实况"的重度用户保留。
>
> **定位**：高德 POI 提供坐标/地址/营业时间/官方评分；美团攻略提供城市/品类/场景维度的**编辑过的好店清单**（数据来源就是大众点评公开数据 + 媒体评测）；小红书提供拍照/氛围/排队/避雷的软信号。三者结合，**80% 场景可以替代 opencli**。

---

## 0. 三档方案（按推荐度排序）

| 档位 | 工具 | 反爬对抗 | 用户介入 | 数据质量 | 适用场景 |
|------|------|---------|---------|---------|---------|
| **主轨 1：高德 MCP POI 详情** | `mcp__amap__*` | 无（官方 API） | 0 | 评分/营业时间/人均/类型 ✅；评价数 ❌ | **必用**：每家候选餐厅的硬信号 |
| **主轨 2：美团攻略 WebFetch** | WebFetch `guide.meituan.com` | 0（编辑内容静态页） | 0 | 城市/品类/场景的**编辑清单**（来自大众点评 + 媒体评测）✅ | **推荐用**：找"某城市某品类的好店" |
| **深度档：OpenCLI + Browser Bridge** | `opencli dianping search/shop` | 0（走真实 Chrome session） | 装扩展 + 扫码 + Chrome 常开 | 真实评价数/排队实况/必吃榜入选/踩雷关键词 ✅✅ | 想要"必吃榜上榜店 + 评价数 + 排队"时才用 |
| ❌ ~~WebFetch M 站~~ | ~~WebFetch `m.dianping.com`~~ | 必败 | — | 评分/评价数被 JS 加密，**几乎拿不到** | **不要用** |
| ❌ ~~Playwright 主站~~ | ~~`mcp__playwright__*` 跑 `dianping.com`~~ | 必败（`verify.meituan.com` 谜题） | — | — | **不要用** |

> **本文件主体只讲主轨 1 + 主轨 2**。深度档（opencli）的安装/用法见 §6（压缩到一节，明确说"非必需"）。

## 1. 主轨 1：高德 MCP POI 详情（零装主力）

### 1.1 怎么调

高德 MCP 已有（**MCP 列表里 `mcp__amap__*` 必须在**——见 `references/setup-guide.md` §1）。三个工具够用：

```
mcp__amap__maps_text_search   # 关键词搜 → 拿 POI 列表
mcp__amap__maps_around_search # 周边搜（按坐标 + 半径）
mcp__amap__maps_search_detail # POI 详情 → 拿评分/营业时间/人均/类型
```

**典型流程**：

```
# 1. 按城市+菜系搜候选
maps_text_search(keywords="火锅", city="成都", types="050101")

# 2. 拿前 5 家的 POI ID → 详情
maps_search_detail(id="B0JA5UF6R*")
→ 返回 {"name": "...", "rating": "4.7", "cost": "120", "opentime2": "...", "type": "餐饮服务;中餐厅;中餐厅", "address": "...", "location": "120.122175,30.253356"}
```

### 1.2 高德 POI 详情能拿到的餐厅硬信号

| 字段 | 字段名 | 用途 | 大众点评对位字段 |
|------|-------|------|---------------|
| **rating** | 评分（0-5） | 基础筛选 | `rating` |
| **cost** | 人均价格（元） | 预算匹配 | `price` |
| **opentime2** | 营业时间（含分时段） | 安排到访 | `open_time` |
| **type** | 分类（"餐饮服务;中餐厅;中餐厅"） | 菜系匹配 | `cuisine` |
| **address** | 详细地址 | 通勤计算 | `address` |
| **location** | "lng,lat" | 通勤计算 | `lng`/`lat` |
| **business_area** | 商圈 | 区域匹配 | `district` |
| **photo** | 头图 URL | UI 展示 | `photo_url` |

**关键差异**：
- ❌ **没有评价数**（高德 POI 详情不返回评价数）
- ❌ **没有排队实况**（高德不做这件事）
- ❌ **没有"是否值得专门去"标记**
- ✅ **评分通常是大众点评的镜像**（高德的数据合作方就有美团），但**评分稳定性不如大众点评**（高德对店家的去重做得粗）
- ✅ **坐标精准**（高德自家制图）

### 1.3 配合 V3 区域匹配的姿势

`references/validation-rules.md` V3 要求"餐厅距当天主区域 < 1.5km"——用高德 POI 的 `location` 字段直接算：

```python
# scripts/validate.py 的 check_v3 实际就这么算
haversine(hotel_coord, poi_coord) <= 1.5  # km
```

**不需要大众点评 district 字段**——高德 `location` + 酒店坐标 直接算直线距离。

### 1.4 局限与补强

| 局限 | 补强方法 |
|------|---------|
| 缺评价数 | 改用"主轨 2 美团攻略"补充"编辑过的好店清单" |
| 缺排队实况 | 小红书搜"<店名> 排队"软信号 |
| 缺踩雷关键词 | 小红书搜"<店名> 避雷" / "踩雷" |
| 评分不稳（同一店评分可能与大众点评有 0.1-0.3 偏差） | V3 阈值放宽到 ≤ 1.5km（V3 已有），不卡评分小数点 |

## 2. 主轨 2：美团攻略 WebFetch（零装补充）

> **新增 v1.2.0**。美团官方 2026 年 4 月新上线的"编辑版"攻略（`guide.meituan.com`），**数据来源就是大众点评公开数据 + 媒体评测 + 真实口碑**。**编辑过的城市/品类/场景清单**，相当于"美团官方筛选过的好店"，**反爬对抗 = 0**。

### 2.1 URL 模式（已验证）

| 内容类型 | URL 模式 | 例子 |
|---------|---------|------|
| 城市美食攻略 | `https://guide.meituan.com/<city_pinyin>/canyin` | `https://guide.meituan.com/shanghai/canyin` |
| 城市住宿攻略 | `https://guide.meituan.com/stay/<city_pinyin>` | `https://guide.meituan.com/stay/wuhan` |
| 城市专题（省钱/季节性） | `https://guide.meituan.com/save-money` 等 | — |
| 城市拼音 | 拼音全小写 | 上海=shanghai、成都=chengdu、北京=beijing |

**8 大城市**有详细餐饮攻略（覆盖 50+ 城市概要）：
- 上海、北京、成都、广州、深圳、杭州、重庆、武汉

**其他城市**的"怎么吃"页可能未上线或很简略，**优先用这 8 个做调研**。

### 2.2 怎么 WebFetch

```
WebFetch: https://guide.meituan.com/chengdu/canyin
```

**返回内容**（典型结构）：

- 城市概览（消费水平、餐饮门店数）
- 品类分类攻略（火锅/烤肉/日料/西餐…）
- 场景攻略（商务/约会/家庭/聚餐…）
- 价格梯度推荐
- 高频关键词（"本地人爱去"、"周末排队"、"必吃榜"等）
- 入选餐厅（编辑挑选，**通常给店名 + 区域 + 价位 + 入选理由**）

> **重要**：编辑页给的是**清单**（通常 5-20 家），不是单店详情。要拿单店坐标/营业时间/官方评分，**回到主轨 1 用高德 `maps_text_search` 查每家店**。

### 2.3 AI 拿到内容后怎么用

**典型工作流**：

```
Step 1: 用户说"成都 3 天，喜欢火锅、避开游客店"
Step 2: AI WebFetch https://guide.meituan.com/chengdu/canyin
        → 拿到"火锅品类"编辑推荐清单（编辑挑过的 5-10 家）
Step 3: AI 对清单里每家店：
        - mcp__amap__maps_text_search(keywords="<店名>", city="成都")
        - 拿 POI ID → mcp__amap__maps_search_detail(id=...)
        - 拿坐标/营业时间/类型 → 算 V3 通勤 → 决定主推 vs 备选
Step 4: AI 调 xhs-skill 搜"<店名> 避雷"/"<店名> 排队" → 软信号补充
Step 5: 输出模板：
        - 主推：<店名>（编辑推荐 + 高德评分 X.X + 营业至 22:00）
        - 备选：<店名>（编辑推荐 + 高德评分 X.X + 距主区域 0.8km）
        - 避雷：<某店小红书多人说不新鲜>
```

### 2.4 局限与应对

| 局限 | 应对 |
|------|------|
| 8 大城市之外内容稀疏 | 改用**高德 POI 详情 + 小红书**，美团攻略只作"国内主流城市"备选 |
| 编辑筛选≠用户偏好 | **只做"候选池筛选器"**，不替代用户最终决定 |
| 没有"实时排队" | 小红书搜"<店名> 排队"+ 标注"工作日午市/周末晚市排队风险" |
| 拿不到必吃榜入选证明 | 主轨 1 拿评分（高德评分 4.6+ 通常对应大众点评必吃榜水平）|
| URL 模式可能变 | **AI 拿到后自己观察**：URL 改了就用新模式（不要硬编码 URL 模板）|

### 2.5 与"软信号"的边界

- ✅ **美团攻略的清单 = 编辑筛选 = 软硬之间**（不是纯软信号，但也不是用户实时评价）
- ❌ **不能**当硬口碑（编辑可能漏选某些好店）
- ❌ **不能**当"必吃榜入选证明"（虽然数据来源里有大众点评，但页面上没标入选）
- ✅ **当候选池筛选器**：从 100+ 候选砍到 5-10 候选

## 3. 主轨 1+2 + 小红书软信号（实际推荐组合）

> **v1.2.0 推荐默认组合**——80% 场景够用，**零反爬对抗、零装、零扫码**。

### 3.1 调研职责分工

| 数据 | 高德 POI | 美团攻略 | 小红书 |
|------|---------|---------|--------|
| 坐标/地址 | ✅ | ❌ | ❌ |
| 营业时间 | ✅ | ❌ | ❌ |
| 类型/菜系 | ✅ | ❌ | ❌ |
| 官方评分 | ✅（与点评有 0.1-0.3 偏差） | ❌ | ❌ |
| 人均价格 | ✅ | 部分（编辑标注的区间） | 部分 |
| **城市/品类筛选** | ❌ | ✅✅ | ❌ |
| **场景筛选**（约会/家庭/商务）| ❌ | ✅✅ | ❌ |
| **入选理由/标签** | ❌ | ✅✅ | ✅ |
| 评价数 | ❌ | ❌ | ❌ |
| 排队实况 | ❌ | ❌ | ✅（近期）|
| 踩雷 | ❌ | ❌ | ✅（专门避雷笔记）|
| 氛围/拍照 | ❌ | 部分 | ✅✅ |
| 工作日适用 | ❌ | 部分 | ✅ |
| 当地人偏好 | ❌ | ✅（"本地人爱去"标签）| ✅ |

### 3.2 组合使用模式

**Step 1 — 美团攻略定方向**：
- WebFetch `guide.meituan.com/<city>/canyin`
- 拿到该城市该品类的**编辑清单**（5-20 家）
- **城市不在这 8 个** → 改用**高德 text_search 按菜系搜**做兜底

**Step 2 — 高德 POI 详情填硬信号**：
- 对清单里每家店 → `maps_text_search` → `maps_search_detail`
- 拿坐标/营业时间/类型/评分
- **算 V3**（餐厅 vs 酒店直线距离 ≤ 1.5km）→ 不达标直接剔
- **算时间**（营业时间 vs 用餐时段）→ 关门了直接剔

**Step 3 — 小红书补软信号**：
- 对剩下 3-5 候选 → 搜"<店名> 排队"/"<店名> 避雷"/"<店名> 拍照"
- 砍踩雷，标排队风险

**Step 4 — 写回模板**（§5）

### 3.3 这个组合的"信号损失"

对比**opencli 深度档**，主轨 1+2 + 小红书少了：

| 缺失信号 | 影响程度 | 替代方案 |
|---------|---------|---------|
| 评价数 | 低 | 城市主流城市不需要评价数来筛 |
| 必吃榜上榜证明 | 中 | 高德评分 4.6+ 通常 ≈ 必吃榜水平 |
| 排队实况（精确到小时）| 中 | 小红书搜"工作日" / "周末"分时段看 |
| 踩雷关键词精确分析 | 中 | 小红书搜"避雷" |
| 用户真实评价数（vs 评价长度/有用数）| 低 | 不影响"能不能进候选池" |

> **80% 场景不影响最终方案质量**。剩下 20% 场景（用户非要"必吃榜上榜店 + 评价 5000+"）才需要深度档。

## 4. 筛选六维（v1.2.0 适配）

每家餐厅候选按 6 维评估：

| 维度 | 评估方式 | 数据来源 | 阈值建议 |
|------|---------|---------|---------|
| **rating** | 评分 | 高德 `maps_search_detail` | ≥ 4.0 基础，≥ 4.5 优秀 |
| **reviews_proxy** | 编辑推荐入选 | 美团攻略清单里 = 入选 | 入选 = 通过 |
| **price** | 人均价格 | 高德 `cost` 字段 | 用户预算 ±20% |
| **cuisine** | 菜系 | 高德 `type` 字段 | 与当天主餐类型匹配 |
| **district** | 区域 | 高德 `location` 算 vs 酒店 | V3 强制 ≤ 1.5km |
| **软信号** | 排队/避雷/氛围 | 小红书笔记 | 无踩雷 + 营业时段不排队 |

> **reviews_proxy 是 v1.2.0 新增**：用"是否被美团攻略编辑收录"代替"大众点评评价数"。**收录 = 编辑认为值得推荐**，比"500+ 评价"更准。

**理想候选**：高德评分 ≥ 4.3 + 美团攻略收录 + 价格在预算内 + 区域匹配 + 小红书无踩雷。

## 6. 深度档：OpenCLI（**v1.2.0 降级为可选**）

> **什么时候才用**：用户**明确要求**"必须大众点评必吃榜上榜 + 评价数 + 排队实况"；**或者**主轨 1+2 的清单里有 5+ 家店拿不定主意，需要**大众点评评价数排序**。
>
> **80% 场景不需要**。**默认不推荐装**——反爬对抗 + 装扩展 + 扫码 + Chrome 常开的成本，对普通旅行者不划算。

### 6.1 装与用（压缩版）

> 完整说明见旧版（git log `v1.0.1` 之前）。**本节只给精简版**——AI 给用户讲方案时**先讲主轨 1+2，讲完用户非要深度档才给本节**。

```bash
# 1. 装 opencli
npm install -g @jackwener/opencli   # Node 21+

# 2. 装 Chrome 扩展（必装，opencli 没有就连不进真实 Chrome）
# Chrome Web Store: https://chromewebstore.google.com/detail/opencli/ildkmabpimmkaediidaifkhjpohdnifk

# 3. Chrome 必须保持运行 + 已登录大众点评
# 验证：opencli doctor → "Extension connected"

# 4. 搜索餐厅
opencli dianping search "烤肉" --city 成都 --limit 5 -f json

# 5. 查店铺详情（带评价数/排队/踩雷关键词）
opencli dianping shop <shop_id> -f json
```

### 6.2 降级路径决策树（v1.2.0）

```
做方案时，餐厅硬信号需要多强？
├── 普通旅行（不需要必吃榜 + 评价数 + 排队实况）
│   └── 主轨 1（高德 POI）+ 主轨 2（美团攻略 WebFetch）+ 小红书软信号
│       ↓
│       候选 5+ 家拿不定主意？
│       ├── 否 → 直接出方案
│       └── 是 → 主轨 2 + 小红书 都筛不动，**才考虑深度档**
│
└── 重度旅行（明确要必吃榜 + 评价数 + 排队实况）
    └── 直接装深度档（opencli + Browser Bridge）
        跳过主轨 2（美团攻略对重度用户也不够）
```

### 6.3 不再走的死路（明确禁止）

- ❌ **不要再尝试 WebFetch 大众点评 M 站**：`m.dianping.com` 反爬比主站更严，且核心数据（评分、评价数、踩雷关键词）经常被 JS 加密拿不到。**直接用主轨 1+2 比硬刚方案 B 省 5-10 倍时间**。
- ❌ **不要再尝试 Playwright 跑大众点评**：`verify.meituan.com` 反爬会识别 Headless Chrome 指纹（`navigator.webdriver=true` + UA 里的 "HeadlessChrome"），跳转谜题页；`playwright-stealth` 插件也救不了（Cloudflare Turnstile 用的是 TLS 指纹 + 行为分析 + 滑块轨迹，不是 stealth 能搞定的）。详见 `references/dianping-research.md` §0 警告。
- ❌ **不要再尝试自己爬主站**：`dianping.com` 有签名 (`_token`) + 行为指纹 + 滑块，CSDN 2026 实战说"自建集群是唯一选择"——普通旅行不值得。

## 7. 常见坑（v1.2.0 适配）

- ❌ 只按评分选店，忽略当天区域（违反 V3）
- ❌ 为单店反向规划半天路线（违反"餐厅是补给不是锚点"）
- ❌ 把小红书种草当硬口碑
- ❌ 漏看营业时间（用高德 `opentime2` 字段，**不要用大众点评数据补**——会引入"主轨混用"的复杂判断）
- ❌ 搜索词过泛，召回一堆游客店（美团攻略清单里**优先选有"本地人爱去"标签的**）
- ❌ 跳过美团攻略直接用高德搜（会召回**大量你没听过的店**，需要 V3 暴力剔——美团攻略先把池子缩到 5-20 家，效率高很多）

## 8. 与 V1-V7 验证的关联

| 验证规则 | 主轨 1+2 + 小红书组合 |
|---------|---------------------|
| **V3 餐厅区域匹配** | 高德 `location` 直接算 vs 酒店坐标（≤ 1.5km 强制）|
| V7 用户禁忌屏蔽 | 美团攻略清单 + 高德 `type` 字段辅助判断（如用户说"不要游客店"，剔没"本地人爱去"标签的）|
