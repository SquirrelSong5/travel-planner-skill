## 🔒 v1.1.0 重要变化：代码级硬约束

> **为什么改**：AI 经常"走过场"——写"✅ 全通过"但没真算过。**v1.1.0 起，验证靠代码不靠自觉**。

**双层硬约束**：

1. **服务端脚本**（`scripts/validate.py`）：AI 必跑的命令，真用 POI 坐标 + 阈值公式算
2. **浏览器端 JS**（`assets/template.html`）：HTML 加载时**强制重算 V1/V3/V4/V5/V6/V8/V9/V10**，**结果覆盖** AI self-report

**AI 写假报告也挡不住**——用户点开 HTML，浏览器会算给你看。

### 总览：哪条脚本能验、哪条必须 AI 跑

| ID | 规则 | Round | 验证方式 | 失败时默认动作 |
|----|------|-------|---------|---------------|
| V1 | 区域一致性 | 1 | ⚙️ **脚本** | 跨区提示换酒店或挪日 |
| V4 | 一日一重预约 | 1 | ⚙️ **脚本** | 拆日或减预约 |
| V7 | 用户禁忌屏蔽 | 1 | 🤖 **AI** | 直接删 |
| V2 | 时间可行性 | 2 | 🤖 **AI 调高德** + ⚙️ 粗算 | 替换/删/挪时间 |
| V5 | 末日返程缓冲 | 2 | ⚙️ **脚本** | 末日去远郊挪前 |
| V8 | MCP 必跑痕迹 | 2 | ⚙️ **脚本** | 重跑高德 direction |
| V9 | 实算 vs 粗算 | 2 | ⚙️ **脚本** | 重跑高德 direction |
| V3 | 餐厅区域匹配 | 3 | ⚙️ **脚本** | 替换餐厅候选 |
| V6 | 户外天气敏感 | 3 | ⚙️ **脚本** | 配室内备选 |
| V10 | 价格溯源 | 3 | ⚙️ **脚本** | 补 price/fare + source |

**AI 调用流程**：

```bash
# 1. 跑脚本（V1/V3/V4/V5/V6/V8/V9 八条 — V7 AI 自填）

# → 拿到 8 条结果

# 2. AI 自己跑高德 route_* MCP（V2）+ 上下文判断（V7）

# 3. AI 把 9 条结果合并为 tripData.validation_report 嵌进 JSON
#    渲染 HTML 后浏览器会**强制重算**这 8 条并覆盖
```

## V1：区域一致性

**规则**：一天内所有 POI 必须落在**同一主区域**。

**主判据（强烈推荐）**：通勤分钟数
- 用高德 `route_transit` 实算每个 POI → 当日主区域中心的通勤时间
- 任一 POI > 20 分钟 → 警告；> 30 分钟 → 失败
- 跨城市尺度下也稳定（东京 5km 走路 1 小时、北海道 5km 开车 10 分钟，分钟数是普适语言）

**快速预筛**：直线距离（仅作预估，不能作终判）
- 主区域用高德 `poi_search keywords="<区域名>"` 取中心经纬度
- 计算每个 POI 到中心点的直线距离
- 任一 POI > 3km → 警告；> 5km → 失败

**失败动作**：
- 选项 A：把远区 POI 挪到相邻的 day
- 选项 B：把相邻 day 的近区 POI 挪过来（合并区域）
- 选项 C：换酒店到新区域的中心

**例外**：
- 多城行程的"跨城日"不算违规，但要在 HTML 标注「跨城日」
- 远郊/一日游日（如富士山、镰仓一日游）按主区域定义本身就不在城市中心，**改用"远郊圈"判定**（POI 之间互相距离 < 10km 即视为同区）
- **`region_flex: true` 字段**：标记当天为"含跨区返程"，脚本自动跳过 V1 区域距离检查。适用场景：
  - 到达日（机场在岛外，酒店在思明区）
  - 白天远郊 + 晚上回市区（如"白天河口湖 + 晚上新宿"）
  - 单日横跨绵延区域（如厦门环岛路 10km+）

## V2：时间可行性

**规则**：相邻 POI 之间的通勤时间 + 上一 POI 停留时间 ≤ 下一 POI 计划开始时间 - 上一 POI 计划开始时间。

**判定方式**：
- 对每一对相邻 POI，用高德 `route_walking` / `route_transit` 实算
- 加上每个 POI 的 `duration_min` 停留时间
- 任一段不够 → 失败

**量化阈值**：
- 单段通勤 > 60 分钟 → 强警告（除非是城际）
- 累计通勤占当天行程 > 50% → 失败（v1.5.0 调整，原 30% 太严）

**例外**：
- **远郊/跨城日不计入累计通勤**（城际新干线、富士山往返这种大块通勤不构成"赶死"信号）
- 仅对"市内单区日"判定 50% 阈值
- **V9 偏差阈值 50%**：walking/transit 模式因绕路/换乘/船等，偏差天然 > 50%，**V9 是设计 bug，详见 P26 修复进度**

**失败动作**：
- 删一个 POI
- 调整时间块（早起 / 晚收）
- 换相邻 POI（同区域内）

**重要**：不能"目测"通勤，必须用 `route_*` MCP 实算。3km 在东京可能 40 分钟，在大阪可能 20 分钟。

## V3：餐厅区域匹配

**规则**：午餐 / 晚餐候选店必须落在当天主区域内（偏离 < 1.5km）。

**判定方式**：
- 每顿餐厅用高德 `geocode` 取经纬度
- 计算到当天主区域中心的距离
- > 1.5km → 警告；> 3km → 失败

**失败动作**：
- 替换餐厅候选（从高德 `poi_search` 在区域内重找）
- 必要时挪整个 day 的主区域

**例外**：跨城日的"路途餐厅"（如新干线上的车站便当）不算违规。

## V4：一日一重预约

**规则**：一天最多 1 个需要提前预约的 POI。

**判定方式**：
- 标记每个 POI 的 `requires_booking: true/false`
- 一天 `requires_booking=true` 的 POI > 1 → 失败

**重预约识别标准**：

国内常见：
- 热门景区分时预约（故宫 / 国博 / 兵马俑 / 莫高窟 / 迪士尼 / 环球影城等）
- 网红餐厅排号（点都德 / 陶陶居 / 文和友 / Shake Shack / Lady M 等）
- 特色体验（汉服 / 茶道 / 中医 / 私厨 / 私人导游 / 包车）
- 沉浸式 / 主题艺术展（teamLab / 不眠之夜 / 各类 IP 展）
- 高端 SPA / 私汤 / 温泉私汤
- 演唱会 / 比赛 / 限定活动 / 大型演出

境外常见（保留）：
- 米其林星级 / 寿司 omakase
- 私人体验课（茶道 / 和服 / 烹饪）
- 团队游 / 包车
- 高端 SPA / 私人温泉

**失败动作**：
- 把其中一个挪到相邻 day
- 把其中一个降级为"免预约替代"（如 teamLab Borderless → teamLab Planets）

## V5：末日返程缓冲

**规则**：末日最后 POI → 酒店 → 机场的时间链上必须有 ≥ 3 小时缓冲。

**判定方式**（v1.5.0 简化版）：
- **实际实现只算"末 POI 结束 → 航班起飞"一步**（不再算酒店→机场通勤时间，因为 V2 已覆盖）
- 末 POI 结束时间 + (酒店到机场通勤) < 航班起飞时间
- **缓冲阈值**（v1.5.0 调整）：< 1.5h 失败 / < 2h 警告 / ≥ 2h 通过
- v1.5.0 之前文档写 < 2h/3h 是过时的

**失败动作**：
- 把远郊 POI 挪到倒数第二天
- 提前最后 POI 结束时间
- 改返程航班 / 火车
- 加一晚分酒店减少末日压力

**重要**：末日必须按航班 / 车次**倒推**，不要按"想去的"正推。

## V6：户外天气敏感

**规则**：户外 / 山岳 / 海边 POI 必须标 `weather_sensitive: true` + 配室内备选。

**判定方式**：
- 标记每个 POI 的 `weather_sensitive: true/false`
- `weather_sensitive=true` 的 POI 必须有 `indoor_backup` 字段（备选室内 POI）
- 缺失 → 失败

**户外识别标准**（**v1.5.0 实际用关键字集合判定**）：
- POI `type: "outdoor"` 字段
- POI `tags` 数组含以下任一关键字：`outdoor` / `户外` / `露台` / `天台` / `观景`

> **v1.5.0 文档与实现不一致**：本文档之前列"山岳/海边/公园"等自然语言类别，但 `validate.py` V6 实际用关键字集合匹配。**AI 应在 tripData POI 上标 `tags: ["outdoor"]` 或 `type: "outdoor"` 触发 V6**。

**失败动作**：
- 补 indoor_backup 字段
- 或挪到非雨天 / 阴天预测的日期

**注意**：不要过度标 —— 城市内步行逛街不算 weather_sensitive。

## V10：价格溯源（v2.1.0 新增）

**规则**：每个 POI、交通段、午餐/晚餐必有调研价 + 合法 `source`；禁止 LLM 猜测。

**判定方式**（`scripts/validate.py`）：
- 缺 `party_size` → 失败
- 每个 `pois[].price` / `transports[].fare` / `meals.lunch|dinner.main.price` 缺对象或缺 `source` → 失败
- `source` ∈ `{ai-guess, memory, ""}` 或不在白名单 → 失败
- `budget_summary.total_min` 与时间轴明细加总偏差 > 15% → 警告

**合法 source**：`amap-mcp` | `amap-rest-api` | `official-site` | `ctrip-webfetch` | `meituan-webfetch` | `computed`（仅 free/步行）

**失败动作**：回 Round 2/3 补 MCP/WebFetch 调研；Step 6 重算 `budget_summary`

详见 `references/price-research.md`。

## V7：用户禁忌屏蔽

**规则**：用户在 Step 1 标记的"不要去的类型"在方案中**完全消失**。

**判定方式**：
- 用户的禁忌列表（如"不要寺庙"、"不要网红打卡"、"不要夜店"）
- 方案中每个 POI 的类型 / 标签
- 任何 POI 匹配禁忌 → 失败

**失败动作**：
- 直接删该 POI
- 必要时替换为同区域不同类型

**重要**：即使该 POI 是"必去"，用户明确说了也不要 —— **用户禁忌优先级最高**。

## 不进 V1-V7 的事项

下列事项虽然重要，但**不属于自动验证范围**，AI 通过其他方式处理：

- 票务 deadline（写进 `prebook`）
- 餐厅口味 / 排队（属于 Step 4 选餐决策）
- 用户审美 / 拍照偏好（属于小红书软信号）
- 季节性活动（属于"先去查"的范畴）

## 无高德 MCP 时的验证降级

高德 MCP 缺失时（用户未配 Key 或降级模式），V1 / V2 / V3 / V5 涉及距离/通勤/实算的规则**无法自动验证**。处理方式：

### 降级动作

1. **明确告知用户**：当前为"通用知识模式"，V1-V5 为 LLM 估算，未实算。
2. **改用 LLM 估算**：
   - V1：基于地名常识判断"是否同区"（如新宿/涩谷/原宿 → 视为同区；新宿/镰仓 → 视为跨区）
   - V2：基于常识估计通勤时间（标注"估算"），不再使用"实算"字样
   - V3：基于餐厅地址常识判断区域
   - V5：基于酒店到机场的常识距离给缓冲估算
3. **在 `validation_report` 标注**：
   ```md
   ⚠️ 降级模式：未实算通勤时间（高德 MCP 未配置），V1/V2/V3/V5 为 LLM 估算，仅供参考
   ```
4. **建议用户回头配高德 Key**（给 5 分钟注册链接，引用 setup-guide.md）。

### 不降级的规则

- V4 一日一重预约：不依赖高德
- V6 户外天气敏感：标记属性即可，不依赖高德
- V7 用户禁忌屏蔽：基于禁忌列表匹配，不依赖高德
